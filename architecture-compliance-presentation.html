01 · The Problem

Three ways drift creeps in

03 / 09
01

New Terraform is created

Fresh resources are added in code — an extra service, a new data store — that the architecture page never mentioned.

+ azurerm_cosmosdb_account
  # not in Confluence
02

Existing Terraform is edited

A SKU is downgraded, a private endpoint removed, HA reduced — small edits that silently contradict a documented decision.

sku_name = "P1V3" → "B2"
  # violates the design
03

Confluence is edited

An architect updates the intended design, but the Terraform is never revisited — now the code lags behind the documented target.

doc: "must be zone-redundant"
  # code not updated

Proposal A · Recommended · Deep dive →

GitHub + Actions

Native to your GitHub DevOps platform
  • ▹Runs on every Pull Request via GitHub Actions
  • ▹Publishes a merge-blocking Check Run (Branch Protection)
  • ▹Auto-opens a Doc-Patch PR when docs are stale
  • ▹Architect sign-off via CODEOWNERS
Proposal B · Alternative

App + Custom Agent
on Azure Foundry

Platform-agnostic, managed agent runtime
  • ▹Web app on Azure App Service receives PR events
  • ▹Custom agent on Azure AI Foundry Agent Service
  • ▹Managed identity, tools & connections built-in
  • ▹Works beyond GitHub — any SCM via webhook

04 · Proof of Concept

How the PoC works today

08 / 09
run-architecture-review.ps1 — deterministic
  1. 1Validate tooling — terraform · az · copilot
  2. 2Enforce current Azure credentials (az account show → inject subscription)
  3. 3terraform init → validate → plan → show -json = plan.json
  4. 4Invoke the skill via the Copilot CLI
plan.json
architecture-drift-review — skill
  1. 1Extract intent model from the design doc
  2. 2Extract reality model from plan.json
  3. 3Compare semantically — ✅ ⚠️ ❌
  4. 4Write the report → architecture-review.md

The script owns the deterministic half; the skill owns the semantic comparison.
